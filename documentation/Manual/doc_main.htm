User's manual

1. About Ethanon Engine

Ethanon Engine is a free 2D game development tool focused on the recent video cards for hardware acceleration and shader effects. It may be used to develop any style of game, such as top-down view RPG's, side scrollers and shoot'em up games. The main goal of this engine is to provide high-quality lighting, shadow and particle effects with an easy to use visual interface and a simple and direct scripting system.

EE is entirely based on 2D sprites. It uses pixel shaders to perform the normal map based lighting and supports any kind of map construction (e.g.: isometric, orthogonal or hexagonal) because it's based on arbitrarily positioned entities instead of fixed tiles.

To control the game flow and handle its events, Ethanon Engine uses AngelScript for scripts, which brings a familiar C++ syntax to the code. The integration with the scripting language is pretty simple and easy to understand. Many features are included in the scripting system, such as full entity manipulation, save and load games with one instruction, video playback support, and more.

 

1.1. File formats

Ethanon Engine supports the following media file formats:

Audio files: OGG, MP3, FLAC, WAV, AIFF, MOD, S3M, XM and IT
Bitmap files: PNG, JPG, BMP, DDS and TGA
Video files: WMV, AVI*, MPG*

*the codec must be supported by DirectShow.

EE also has its own file formats:

Project: ETHPROJ
Entities: ENT
Particle Effects: PAR
Scenes: ESC
 Script code: AS

 

2. 2D Entities

Every scene in Ethanon Engine is based on 2D entities. These entities are the objects and elements from the game, such as walls, pillars, crates, characters, space ships, enemies, candles, torches or even the floor tiles. Each entity may consist of:

This bitmap will inform the renderer what the entity surface is to look like. A normal map is necessary for lighting. Entities without a normal map will not be lit.

Once placed in the scene, every single entity is assigned an identification number (ID number). Many entities in a scene will probably share or have several properties in common, e.g.: the same original file name, the same sprite file, etc. but never the same ID number.

 

2.1. Entity types

It is important to inform the engine which type each entity is so that it can know how to render and how each one should behave. There are three basic entity types: "vertical", "horizontal", and "layerable":

 

Horizontal entity: this type of entity should always be assigned to floor tiles. It is important to set every ground entity as horizontal because from a top-down perspective they will always lay on the "bottom layer". The engine must know that.

 

Vertical entity: entities that visually* have three-dimensions. Examples of vertical entities: crates, pillars, walls, barrels, characters, explosions, fire, smoke, torches, trees, bushes, etc.
*technically, the entity will never be a true three-dimensional object since Ethanon Engine is based on 2D sprites, but they may represent entities in a three-dimensional space.

  Layerable: these entities will be drawn and ordered by their own user-defined depth value. Depth values may range between 0 (furthest) and 1 (closest). For instance: an entity whose depth equals 0.5 will always lie behind an entity whose depth equals 0.9. Layerable entities are useful for 2D games that do not use a top-down view, like side scrollers (the landscaping and backdrop images may use depth 0, the tiles may use 0.5, etc). The entity depth may be defined in the entity editor.

All entities may be rotated except vertical entities.

 

2.1.1. Temporary entities

Entities that have only finite particle systems and no sprite will be considered as temporary. Which means that they will be automatically deleted as soon as all particles systems are over.

Finite particle systems are the ones whose repeat times are greater than 0. When the repeat count is zero, the particle cycles will stop only if the entity is deleted.

 

2.2. Lights and entities

There are two ways of lighting a scene:

1. Lights attached to entities: lights attached to entities may look like "natural" lights such as torches, candles, fire, lamps, etc. or they may have no explicit light source if the entity has no sprite. The entity may have no sprite or particle effects but only a light source; it will still count as an entity but the user will only see the light source.
An Entity light will always follow the entity if it moves through the scene. And they may be static or dynamic depending on their parent entity configuration.

2. Odd lights: lights without parent entity. Can only be placed in the scene through scripting (with the AddLight function) and are immediately erased after being rendered. To keep these lights in the scene they must be added at every frame.
Odd lights are always dynamic, never static.

 

2.3. Static and dynamic entities

Entities may be presented in two different modes, no matter what type it is, it can either be set as static or as dynamic. The main difference between them is that static entities are much faster to render. The more static entities the scene has, specially those with their own light sources, the faster the scene will be rendered. The reason why is described below:

Dynamic entities: all lighting and shadowing of dynamic entities will always be performed in real-time. Which means that for every light (dynamic lights or static lights) close to the dynamic entity, the shaders will be executed to compute how these lights affect the entity.

Static entities: instead of computing every nearby light source, static entities will have all static lights pre-rendered to it as a lightmap. This is how static lights (light sources that come from static entities) and static entities are much faster. In these cases, the lighting effect comes almost for free, the only cost is to apply the lightmap as a multitexture.
But be careful, dynamic lights will light static entities dynamically. It's important to remember that "free lights" only happen for static entities lit by static lights (lights which parents are static).

Remember that if an entity has a light source attached to it, the light source will be either static or dynamic depending on its parent configuration.

 

2.4. Entity space

Although this is a 2D game engine, the entities have a three-dimensional positioning system. The world-space coordinates in Ethanon Engine are very similar to a regular 2D screen projection. It has its origin at the upper left corner of the screen, the X-axis points to the right and the Y-axis points downwards. The only difference is that there is a Z-axis that points upwards, it gives the impression that the entity is floating because of the viewer's perspective.

3. Ethanon Editor

With the edition program (Ethanon Editor.exe) developers are allowed to visually create, configure and edit the entities, as well as particle systems that can be then attached to the entities and also create the whole scenes and see what they are going to look like in the game in real-time.

In order to make the development easier and faster, Ethanon Editor is divided into three main tools: ParticleFX Editor, Entity Editor and Scene Editor. These tools can be switched during the execution of the Ethanon Editor by pressing the TAB key. It allows the developer to quickly swap between tools without interfering with their individual execution. For instance, if during the scene creation the user is not happy with the appearance of a particular entity, it can immediately switch to the Entity Editor, open that entity file, make the desired changes, save it, and go back to the Scene Editor where it can be replaced.

 

3.1. Input fields

The Ethanon Editor tools have several numerical input fields with which the user can control values that will define different behaviors of its elements. These fields normally have a description and an input field. After selecting any of these fields by clicking them, there are two different ways of editing these values:

-Keyboard: numerical fields can be edited through the regular keyboard keys.
-Mouse: it is also possible to gradually increase or decrease any of these values by scrolling the mouse wheel up or down.

4. Ethanon scripting

All entities, particles and scenes can be visually created and edited with the Ethanon Engine Editors, but more programming is needed. For game development, programming tools are essential, because they make everything happen. There must be a way to handle events and flow control, for example: moving and animating a character when the player uses his keyboard, controlling monsters and NPCs, going to the next level, saving and loading games, etc. For that purpose, EE uses the AngelScript script language. As described in its documentation:

The script language is based on the well known syntax of C++ and more modern languages such as Java, C#, and D. Anyone with some knowledge of those languages, or other script languages with similar syntax, such as Javascript and ActionScript, should feel right at home with AngelScript. Contrary to most script languages, AngelScript is a strongly typed language, which permits faster execution of the code and smoother interaction with the host application as there will be less need for runtime evaluation of the true type of values.

This section of the manual assumes that the reader is already familiar to C, C++, Java, C#, D or PHP (intermediate C++ experience is recommended), and it intends to introduce the reader to the scripting language, showing the main differences between AngelScript and C++ and how the engine reads and executes the scripts. From this point, it is recommended that users who have never used AngelScript take a look at its manual: http://www.angelcode.com/angelscript/sdk/docs/manual/index.html. If you decide to read the manual, go directly to "The script language" section.

4.2. How it works

Every Ethanon Engine script code must start from a main.angelscript code file that must be located at the project's main path. It's necessary to run the 'machine.exe' executable from exactly the same path as the main.angelscript file in order to compile and execute it. It will automatically load the script code, compile it and start the game.

Ethanon *.angelscript script files can be edited with the text editor included within the EE SDK, SciTE. SciTe may also be used to execute the compile/build command (F7 key) or even test the game application (F5 key) without leaving the script editing interface.

The program will always start from the main function:

main.angelscript:

void main()
{
    // loads my_scene.esc
    LoadScene("scenes/my_scene.esc", "", "");

    // hides the mouse cursor
    HideCursor(true);

    // set screen options
    SetWindowProperties(
        "Ethanon Engine", // window title
        1024, 768,        // screen size
        true,             // true=windowed / false=fullscreen
        true,             // true=enable vsync false=disable vsync
        PF32BIT);         // color format
}

Using the SciTE version that comes with the SDK to edit EE's .angelscript script codes will highlight every AngelScript reserved word, EE's native global function names, objects and constants.

Remember that every main.angelscript file must have a main() function, and this function must be void and should not have parameters.

LoadScene is a global function that loads the scene whose file name is informed in the first parameter. The code below will load the 'my_scene.esc' scene file and start rendering it in a regular game loop. The next two empty strings will be explained in the next section.

The HideCursor and SetWindowProperties functions are EE's native globals as well. They may be called to setup custom system and window options, but are both optional. If the function SetWindowProperties is ommited in the main() function, the core engine will use its internal default options. Also, this function may be called at any time to change windows states such as changing screen size or toggling between windowed and full screen modes.

 

4.2.1. Using pre-compiled byte code

When publishing a game it is not always a good idea to release its source code with it. Games made with Ethanon Engine don't need to provide the main.angelscript file (or any other AngelScript source files included, like eth_util.angelscript) if the publisher doesn't want to. EE may also load pre-compiled byte code instead of pure text source-code.

Every time machine.exe is executed, it automatically compiles the main.angelscript file to byte code and saves it to the game.bin file in the project's root. The game may be published without having its source-code revealed if the main source file is removed. When the machine can't find the main.angelscript file it will automatically search for the game.bin file, and if it's valid, it will load the byte code instead of recompiling the program. For that reason, it's important to make sure that the machine.exe had been executed after the game code is finished in order to compile its lastest version.

Before publishing the game, make sure that the machine.exe version that will run the byte code is the same one that generated it, otherwise the application will probably crash.

 

4.2.2. onSceneLoaded, onSceneUpdate and onResume functions

There are three callback functions that may be related to a scene: onSceneLoaded, onSceneUpdate and onResume functions. onSceneLoaded and onSceneUpdate functions are the most used ones.

onSceneLoaded functions will be executed only once immediately after the scene is loaded but before it gets rendered for the first time. These functions are useful for procedures like adding NPC's randomly in the scene before a level starts, or adding custom data to the entities and much more.

Loop functions will be executed at the end of every repeat of the main game loop. This is important to manage menus, control entities or general game events (e.g.: checking when all conditions to finish a level are true), controlling ambient sound effects or whatever the game needs.

To assign these scene callback functions to a scene it is necessary to specify the exact names of the functions when calling the LoadScene function, for example:

main.angelscript:

void main()
{
    // loads my_scene.esc         onSceneLoaded onSceneUpdate
    LoadScene("scenes/my_scene.esc", "NewGame", "GameLoop");

    // hides the mouse cursor
    HideCursor(true);

    // set screen options
    SetWindowProperties(
        "Ethanon Engine", // window title
        1024, 768,        // screen size
        true,             // true=windowed / false=fullscreen
        true,             // true=enable vsync false=disable vsync
        PF32BIT);         // color format
}

void NewGame()
{
    // do something...
}

void GameLoop()
{
    // do something...
}

Notice that LoadScene has the function names as second and third parameters. NewGame is going to be the onSceneLoaded function and GameLoop is the onSceneUpdate function. If these parameters aren't empty strings, the core engine will search for the functions with these exact names. If they cannot be found, the program will exit and output an error message.

If both or any of these parameters are set as empty strings, the engine will not execute any function, e.g.:

    // this one won't have a onSceneLoaded function
    LoadScene("scenes/my_scene.esc", "", "GameLoop");

or

    // this one won't have a onSceneUpdate function
    LoadScene("scenes/my_scene.esc", "NewGame", "");

or even

    // this one won't have any function
    LoadScene("scenes/my_scene.esc", "", "");

onSceneLoaded and onSceneUpdate functions must be void and cannot have parameters.

After loading the first scene and quitting the main() function, the application flow control will be up to the onSceneUpdate function. The following example loads the first scene and while executing the first scene onSceneUpdate function, checks the space key and load the next scene on key hit:

main.angelscript:

void main()
{
    // loads my_scene.esc
    LoadScene("scenes/my_scene.esc", "", "GameLoop");

    // hides the mouse cursor
    HideCursor(true);

    // set screen options
    SetWindowProperties("Ethanon Engine", 1024, 768, true, true, PF32BIT);
}

void GameLoop()
{
    // checks if the space key is pressed
    ETHInput @input = GetInputHandle(); // don't worry, these lines will be
    if (input.KeyDown(K_SPACE))         // explained later
    {
        // loads the next scene
        LoadScene("scenes/level_two.esc", "", "");
    }
}

Other onSceneLoaded and onSceneUpdate functions could be assigned to the next level. A different scene may either share the same callback functions or use exclusive ones.

onResume functions may also be assigned to a scene and will be called whenever the aplication is paused by the system and all its resources (textures and audio samples) deleted (for instance, it happens on Android, when the user closes the application window and then reopen it later). Although Ethanon engine realoads all resources automatically, the user might want to reload everything at once. Read section 4.3 for LoadScene variations and how to set onResume functions.

4.2.3. Entity exclusive callbacks

onSceneLoaded and onSceneUpdate functions are not the only sort of callback functions available. Every different entity may have its own callback function. Entity exclusive callback functions are automatically assigned to an entity whenever the engine finds a function named ETHCallback_entity_original_filename. ETHCallback must always be the prefix for this kind of callback.

The core engine will search for a callback function for every entity that is loaded from the .ESC file or dynamically added to the scene (through script code).

Consider the following example:

main.angelscript:

void main()
{
    // loads raccoon_city.esc
    LoadScene("scenes/raccoon_city.esc", "", "");

    // hides the mouse cursor
    HideCursor(true);

    // set screen options
    SetWindowProperties("Ethanon Engine", 1024, 768, true, true, PF32BIT);
}

// This function will be automatically assigned and executed for the zombies
// No need for manual setup
void ETHCallback_zombie(ETHEntity @thisEntity)
{
    // controls the zombie:
    // -search for fresh brains
    // -die for good when seriously injured
}

Considering the example above, if the scene 'raccoon_city.esc' contains 20 entities with an original name 'zombie.esc', the function ETHCallback_zombie(ETHEntity@) function will be executed 20 times.

All callback functions must have as parameter an ETHEntity@ object handle as shown above, that object handle will always point to the current entity. If this function is executed 20 times per frame (to process all the 20 zombies), every time this function is called in a frame, thisEntity will point to a different zombie. Remember that an object handle declaration follows the @ operator in AngelScript.

Entity exclusive callback functions are great if the game needs entity specific behaviors. The ETHEntity object has several methods to control every important property of the entity (position, angle, animation and much more).

It is also possible to assign specific callback functions based on entity ID, by using the ID number instead of file name:

main.angelscript:

void main()
{
    // loads raccoon_city.esc
    LoadScene("scenes/raccoon_city.esc", "", "");

    // hides the mouse cursor
    HideCursor(true);

    // set screen options
    SetWindowProperties("Ethanon Engine", 1024, 768, true, true, PF32BIT);
}

// This function will be automatically assigned and executed for entity 87
// No need for manual setup
void ETHCallback_87(ETHEntity @thisEntity)
{
    // do something with entity 87
}

If our scene has an entity with an ID equal to 87, the function ETHCallback_87 will be assigned to it, otherwise nothing will happen.

4.2.5. Runtime generated scenes

If you want to generate your game scene during runtime instead of loading a prebuilt one, you may set the string "empty" or an empty string "" as scene file name:

    // loads an empty scene
    LoadScene("empty", "NewGame", "GameLoop");

or

    // does the same as the example above
    LoadScene("", "NewGame", "GameLoop");

By choosing an empty scene, the user tells EE to create a scene without loading entities from an .ESC file.

For dynamically generated scenes, it is recommended to use only dynamic lighting by calling EnableLightmaps(false) in order to disable all static lighting. The only drawback is that the shadows won't look as good as they look as static shades.

 

4.4. Custom variables

It is possible to store custom data which can be retrieved at anytime for an entity. It allows entities to have much more specific behavior. For example, NPC's HP, projectile directions (each one should have a specific direction), RPG character's status such as strength, stamina, agility, etc.

4.4.1. Adding custom variables

Custom variables may either be added through the entity and scene editors or through code.

Every custom data added to an entity must have a name, just like a regular variable. The methods ETHEntity::SetFloat, ::SetInt, ::SetUInt and ::SetString (among others, see the ETHEntity object reference for more information) may be used to add custom data or to update existing values:

Custom data sample:

// Retrieves a handle to our character
ETHEntity @mainCharacter = SeekEntity(nID);

// Add custom data to it
mainCharacter.SetString("class", "mage");
mainCharacter.SetFloat("speed", 100.0f);
mainCharacter.SetInt("hp", 60);
mainCharacter.SetUint("maxHp", 70);
mainCharacter.SetVector2("direction", vector2(20, 40));

// Now hp equals 70. Instead of adding new data,
// it will just update it because "hp" already exists.
mainCharacter.SetInt("hp", 70);
// Will fail because speed already exists and is not and int
mainCharacter.SetInt("speed", 150);

In order to retrieve data, use ETHEntity::GetFloat, ::GetUInt, etc.

 

4.4.2. Retrieving custom variables

To check if a custom variable exists or even to find out of what type it is, use the CheckCustomData method. The methods ETHEntity::GetFloatData, ::GetUIntData, ::GetIntData and ::GetStringData can be used to retrieve custom data from entities:

Checking data:

// Retrieves a handle to our character
ETHEntity @mainCharacter = SeekEntity(nID);

// Checks if "class" exists
DATA_TYPE dt = mainCharacter.CheckCustomData("class");

// if class doesn't exist
if (dt == DT_NODATA)
{
    print("No data here\n");
}
else if (dt == DT_STRING) // if it exists and it is a string, print it
{
    print(mainCharacter.GetStringData("class"));
}

If the ETHEntity::Get____Data methods try to retrieve an inexistent variable, it will always return 0 or an empty string. When you are not sure a certain variable exists or not, always test it with ETHEntity::CheckCustomDatabefore retrieving.

 

4.5. Entity arrays

The entity exclusive callbacks are useful for programming specific behavior for every different entity in scene, but not as useful for creating entities that interact with each other. Let's take a real-time strategy game (RTS) as an example: while allied units are walking to their destiny, it's an important task to check whether they are "walking through" each other or not. Also, when the player is using the mouse to select some of his units, it is difficult to find which units are inside the selection area. Entity exclusive callbacks are also not efficient on massive operations such as processing two fighting armies in games as the above mentioned RTS.

For that reason, sometimes it will be necessary to store entities in a quick access array that could simplify management of multiple objects. In Ethanon Engine this can be done with the ETHEntityArray object. This object is very similar to the std::vector template, in fact, it is a kind of std::vector wrapper that provides the most important methods such as push_back, size and the [] operator.

The most powerful aspect of the ETHEntityArray object is that the GetEntityArray function may build and return an array containing all entities of certain name in scene.

 

4.5.1. Filling the array

To retrieve an array, use the GetEntityArray function:

ETHEntityArray bombs;
GetEntityArray("bomb.ent", bombs);

Now bombs stores an array that contains every bomb.ent entity in scene. The index [] operator for this object is overloaded in order to provide direct access the the ETHEntity@ handle.

It is also possible to fill an array with the push_back method, as shown below:

ETHEntityArray party;
party.push_back(SeekEntity("knight.ent"));
party.push_back(SeekEntity("mage.ent"));
party.push_back(SeekEntity("rogue.ent"));
party.push_back(SeekEntity(nDruidID));

The += operator is also overloaded. It may be used to concatenate two arrays.

 

4.5.2. Reading the array

The ETHEntityArray object allows direct access to the ETHEntity@ handle through index [] operators just like in C++:

for (uint t=0; t<bombs.size(); t++)
{
    // Checks if the current handle is valid
    // before doing anything
    if (bombs[t] is null)
        continue;
    if (!bombs[t].IsAlive())
        continue;

    // moves the entity to the right
    bombs[t].AddToPositionXY(vector2(3.0f,0.0f));
}

See the ETHEntityArray section to learn more about this object.

window

5. Scene buckets

The scenes are partitioned into several buckets for culling and faster collision detection. The default bucket size is 256x256 pixels, but these values may be customized. The buckets consist of rectangular areas that own the entities inside their boundaries. The buckets may be identified by their position on a 2D board (the keys). Consider the image below:

The buckets are identified by their two-dimensional key. The bucket (1,1) contains four barrels, the bucket (-1,0) contains two pillars, etc. Notice that the entities in the buckets (-1,1) and (0,0) exceed their bucket boundaries, but they are owned by the bucket in which their origin is located.

The core engine automatically sorts all entities and places them into their respective bucket, so the programmer must not worry about managing it. However, the programmer may take advantage of this system to optimize its own code. For instance, if the game requires collision detection testing between a character and other scene elements, it is possible to retrieve an array containing only the entities located in the bucket where the character is, in order to check collision only against these buckets instead of checking it against the whole scene or other unnecessary entities.

 

5.1. Creating entity arrays from buckets

To get access to all entities in a bucket, use the global function GetEntitiesFromBucket:

ETHEntityArray bucketArray;

// Parameters: bucket, destination array
GetEntitiesFromBucket(vector2(1,1), bucketArray);

for (uint t=0; t<bucketArray.size(); t++)
{
    bucketArray[t].SetColor(newColor);
}

This function will add a reference to every entity in bucket (1,1) to the array.

It is also possible to hold two or more buckets in a single array by using the += operator to concatenate them:

ETHEntityArray bucketArray;
GetEntitiesFromBucket(vector2(1,1), bucketArray);

ETHEntityArray otherArray;
GetEntitiesFromBucket(vector2(1,2), otherArray);

bucketArray += otherArray;

otherArray.clear();

bucketArray will now hold all entities in the buckets (1,1) and (1,2).

The global function GetBucket may be used to easily find out to which bucket a point in space belongs:

ETHEntityArray bucketArray;
const vector2 v2Bucket = GetBucket(monster.GetPositionXY());
GetEntitiesFromBucket(v2Bucket, bucketArray);

Or you can use the ETHEntity::GetCurrentBucket method to quickly find which bucket holds the entity:

ETHEntityArray bucketArray;
const vector2 v2Bucket = monster.GetCurrentBucket();
GetEntitiesFromBucket(v2Bucket, bucketArray);

 

5.2. Changing bucket size

The only way to manipulate the bucket size is before loading a scene. The LoadScene function is overloaded to receive the bucket size from its last parameter. The minimum width and height for the buckets is 128 pixels.


LoadScene("dungeon.esc", "", "GameLoop", vector2(256,128));
 

dungeon.esc will now be partitioned into 256x128 buckets.

 

5.3. Optimizations and visual artifacts

By default, the core engine doesn't draw ONLY the buckets that are currently visible. Due to that fact that entities might exceed their bucket seams, the surrounding buckets must be included to the render list to avoid undesired visual artifacts, as the image below shows:

In this example, if an entity in a yellow bucket exceeds its seams and partially appears in a blue bucket, it will be drawn normally. This technique avoid visual artifacts, since entities won't mysteriously disappear as their buckets slide away. However, as a drawback, this technique might render many entities that are outside the screen field. For that reason, EE allows the user to disable this technique, and render only visible buckets:

// Disable border buckets drawing. Set 'true' to enable it again
SetBorderBucketsDrawing(false);

Turning off the border bucket drawing is an important optimization on the game. A good level design is very important because if the scene doesn't have big entities that exceed their bucket seams, the border drawing may be turned off and the frame rate may increase drastically.

 

5.4. Remarks on bucket usage

-The Scene Editor has 512x512 pixels buckets instead of the default 256x256.
-For correct lighting and culling, the entities in scene shouldn't be larger than the buckets. For example: if your bucket dimension equals 128x128 pixels, the entity sprites in this scene shouldn't exceed that size. Ethanon Engine won't return any error messages if the scene doesn't follow these rules, but the end-user will probably experience inadequate lighting and culling artifacts.
-Hold F2 on debug mode to show the buckets. Remember that for performance and safety reasons, the bucket size in the editor is locked at 512x512 pixels.
-An entity may intersect two or more buckets at the same time, but it will only be placed in the bucket where its origin is. The programmer must keep that in mind while performing operations such as collision detection, and maybe even including to the test the buckets around the main bucket to avoid disconnecting entities.
-Although the bucket key is based in a floating point vector (vector2), its identification must always be integer. E.g. Right: (40,3), wrong: (10.5f,0.333f).
 

6. Important remarks

6.1. Engine versions

  • If you are reading this text on-line, make sure you have the last Ethanon Engine binaries.
  • Always read the change log after updating your Ethanon Engine version.
  • Your old project files (machine.exe, GameSpace.dll, etc.) won't be automatically updated when you install a new version of Ethanon Engine. To update your project, just open the project manager in the Ethanon Editor, choose the "New project..." option and overwrite your old *.ethproj file. Don't worry, you source code will remain intact, only .EXE and .DLL files will be replaced.

6.2. Optimization tips

  • Minimize the number of horizontal entities (floor, ground, etc.) by using bigger bitmaps when possible. It'll make your game run faster.
  • Use sound effects wisely. Remember that the same sound sample can't be played more than once at the same time and they can only be positioned in one place at a time. So if two particle systems share the same sound effect and they are placed in two different places at the same time, they will swap sides quickly and the result may sound awkward. If you want sound effects to be played at the same time, consider using two or more similar sound effects (e.g.: two slightly different explosion samples).
  • Avoid using the SeekEntity function too many times during the loop because it may become slow. Prefer manipulating entities through exclusive callback functions. Global processes with loops that scan all entities must be avoided and most of them can be replaced by callback functions or entity arrays.
  • Using aligned power-of-two sized ground tiles (32x32, 64x64, 128x128, 256x256...) will make the occlusion system run much more efficiently.
  • A good level design is very important because if the scene doesn't have big entities that exceed their bucket seams, the border drawing may be turned off and the frame rate may increase drastically.

6.3. Safety tips

  • The ETHEntity object is reference counted. It means that as long as the application keeps a handle (ETHEntity@) alive, the entity will still be allocated, even if it has already been deleted from the scene. It will only be freed when all handles are destroyed, so avoid having more than one handle at the same time. Global handles are the most unsafe.
  • Avoid holding global entity handles (ETHEntity@). Prefer holding handles in the ETHEntityArray object.
  • The console window will output any relevant information about the program. Pay attention to it specially in case of an abnormal functioning.
  • All boolean functions will return false if they fail. Error checking is always a good habit.

6.4. Miscellaneous tips and remarks

  • If the particle system in the first slot (A) is not infinite, its activity will control the entity's light brightness.
  • Whenever a spriteless entity has both particle systems finished (if they are not infinite) it will be automatically deleted.
  • The LoadScene function will not immediately load a new scene. Instead, it will request the change as shown in the main loop diagram.
  • Once an entity is placed into the scene using the Scene Editor, its original name will remain the same even if the original file is renamed. In other words, the method ETHEntity::GetOriginalFileName will always return its name as it was at the moment it was placed in the scene.
  • The default color keying invisible pixel color for sprites is FF00FF (R-255, G-0 and B-255). So if you're not using a file format without pixel transparency value, use this background color.
